# Feature: Internationalization (i18n)

## Summary
Multi-language support for `apps/web` using **next-intl** with **locale-prefix URLs**: `/en/...` and `/no/...`. Default locale is `en` (`proxy.ts` redirects `/` to `/en`). Server Components use `getTranslations`; Client Components use `useTranslations` with messages from `NextIntlClientProvider`.

## Status
status: complete

## Scope
- **App(s):** web only
- **Entry points:**
  - [`apps/web/proxy.ts`](../../../apps/web/proxy.ts) — Next.js 16 **proxy** (replaces `middleware.ts`); `createMiddleware(routing)` from next-intl; matcher excludes `api`, `_next`, `_vercel`, static files
  - [`apps/web/app/[locale]/layout.tsx`](../../../apps/web/app/%5Blocale%5D/layout.tsx) — root `<html lang={locale}>`, `setRequestLocale`, `NextIntlClientProvider`
  - [`apps/web/app/[locale]/page.tsx`](../../../apps/web/app/%5Blocale%5D/page.tsx) — home page; SSR translations

## Key Files
| File | Role |
|------|------|
| `apps/web/i18n/routing.ts` | `defineRouting({ locales, defaultLocale })` — single source of truth |
| `apps/web/i18n/request.ts` | `getRequestConfig` — loads `messages/{locale}.json` per request |
| `apps/web/i18n/navigation.ts` | Locale-aware `Link`, `redirect`, `useRouter`, `usePathname`, `getPathname` |
| `apps/web/proxy.ts` | Intercepts requests; enforces locale segment (Next.js 16 convention) |
| `apps/web/next.config.js` | `createNextIntlPlugin('./i18n/request.ts')` |
| `apps/web/messages/en.json` | English strings |
| `apps/web/messages/no.json` | Norwegian strings |

## Locales
| Code | Language |
|------|----------|
| `en` | English (default) |
| `no` | Norwegian |

## API / Usage (patterns)
- **Server Component:** `import { getTranslations } from 'next-intl/server'` then `const t = await getTranslations('home'); t('key')`
- **Client Component:** `import { useTranslations } from 'next-intl'` then `const t = useTranslations('home')`
- **Links / navigation:** Import `Link`, `useRouter`, `usePathname`, `redirect` from `@/i18n/navigation` — **not)** from `next/link` or `next/navigation` for locale-correct URLs
- **Route structure:** All user-facing pages live under `app/[locale]/...`

## Data / config
- No database. Messages are static JSON only.

## Environment Variables
_None required for i18n._

## Dependencies on Other Features
- Typography / Tailwind: body still uses `font-ringside-narrow`; CSS modules reference `var(--font-ringside-narrow)` from shared Tailwind theme

## Key Decisions & Gotchas
- **No root `app/layout.tsx`:** Only `app/[locale]/layout.tsx` wraps `<html>` / `<body>`; every page must be under `[locale]`.
- **Proxy vs middleware:** Next.js 16 renamed the convention to [`proxy.ts`](https://nextjs.org/docs/app/api-reference/file-conventions/proxy). next-intl still exports `createMiddleware` from `next-intl/middleware`; wire it as `export default createMiddleware(routing)` in `proxy.ts`.
- **Dev Turbopack panics:** If `next dev` with Turbopack logs `FATAL: An unexpected Turbopack error`, use `bun run dev` (uses `--webpack`) or `bun run dev:turbo` to try Turbopack again after upgrades.
- **Adding a locale:** Add code to `i18n/routing.ts` `locales` array, add `messages/{code}.json`, rebuild.
- **Norwegian:** Uses Bokmål-style UI strings; adjust copy in `messages/no.json` as needed.
